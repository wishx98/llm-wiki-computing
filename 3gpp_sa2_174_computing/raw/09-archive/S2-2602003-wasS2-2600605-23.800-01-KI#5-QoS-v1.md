**1. Introduction**

The current 5G QoS framework relies on static QoS Flow Identifiers (QFI) which are insufficient for highly asymmetrical 6G traffic profiles, such as Generative AI and Mixed-Media services. These services suffer from **"Encryption Blindness,"** where end-to-end encryption (TLS/QUIC) masks application intent, such as distinguishing an AI text prompt from a heavy image upload.

Generative AI and XR traffic break legacy QoS assumptions. One-size-fits-all network fails modern traffic patterns. The RAN treats all these patterns as identical generic data because the application intent is hidden behind encryption.

**![](data:image/png;base64...)**

Furthermore, a **"Predictive Gap"** exists because the network currently reacts only after traffic arrives in the buffer. For Generative AI, the network is "blind" to the significant downlink response (Text trickle vs. Image burst) while the server is still processing the uplink prompt.

The 6G System must support emerging applications like **Generative AI** and **XR**, which introduce highly asymmetrical traffic patterns. Current frameworks are limited by **"Encryption Blindness"**, where the network cannot distinguish between a low-priority background packet and a latency-critical application burst (e.g., an AI prompt or game input) within the same encrypted tunnel.

This contribution proposes the **Context-Aware Header (CAH)** to bridge the gap between application intent and network resource management. This paper proposes solution to resolve the following open issues in KI#5:

**Bullet 1:** Required functionality for emerging traffic (AI/ML).

**Bullet 4:** Network and application QoS collaboration to improve awareness of traffic needs and what the network can provide.

**Bullet 6:** Providing QoS differentiation for application traffic with multiplexed media flows within a single encrypted tunnel.

## 6.X Solutions to KI#5

### 6.X.Y Solution Variant#5.6: Context-Aware Header (CAH) for Predictive QoS

#### 6.X.Y.0 High-level solution Principles

Support Context-Aware Header (CAH), a lightweight 16-bit adaptation layer inserted between the User IP layer and the 3GPP SDAP layer. It provides a standardized signaling mechanism that remains visible to the network after PDCP decryption without compromising end-to-end payload privacy.

Head-of-Line Determinism: the CAH is placed at the "head-of-line" (immediately after the SDAP header). This enables O(1) zero-search parsing by gNB hardware, critical for 0.5ms TTI deadlines.

Uplink-to-Downlink Prediction: The UE provides a "Workload Tag" in the uplink request that acts as a Downlink Predictor. This signals the *expected* traffic pattern of the response (e.g., text, image, or video) before the server even begins transmission.

Dual-Node Visibility:

RAN (gNB): Performs a non-destructive read of the CAH to optimize scheduling (e.g., extending C-DRX for AI text or pre-allocating large grants for an image burst).

Core (UPF): Extracts metadata for intelligent routing (e.g., Edge/MEC offload) and strips the 2-byte header before the packet leaves the 6G System. This ensures the packet appears as standard IP to the external internet, avoiding "Middlebox Rejection".

#### 6.X.Y.1 Description

This solution introduces a Context-Aware Header (CAH), a lightweight 16-bit signaling layer inserted strictly between the User IP layer and the 3GPP SDAP layer.

Key Architectural Principles include:

System Enablement and Safety: The insertion of CAH is negotiated at the NAS level during PDU Session Establishment and activated via RRC Reconfiguration, ensuring it is only used by authorized UEs in supporting networks.

Capability Negotiation: CAH Support is negotiated during PDU Session Establishment via ePCO to ensure only authorized UEs and supporting networks utilize CAH.

RRC Activation: The gNB configures the UE via RRCReconfiguration to the UE to enable CAH insertion on specific Data Radio Bearers (DRB) and configures the necessary parsing offsets.

Uplink-to-Downlink Prediction: By tagging the uplink request (e.g., "Image Generation"), the UE enables the gNB to proactively pre-allocate Downlink Resource Blocks for the anticipated response before the server finishes processing. The UE provides a "Workload Tag" in the uplink request that acts as a Downlink Predictor. This signals the *expected* traffic pattern of the server's response (e.g., text, image, or video) before the server begins transmission.

Dual-Node Visibility:

RAN (gNB): Performs a non-destructive read of the CAH to optimize scheduling (e.g., pre-allocating grants for an expected image burst).

Core (UPF): Consumes the metadata for intelligent routing (e.g., Edge offload) and strips the header before the packet leaves the 6G System, ensuring compatibility with standard internet middleboxes.

Head-of-Line Determinism: Unlike transport-layer options, the CAH is placed at the "head-of-line" of the SDU, enabling O(1) zero-search hardware parsing at the gNB.

Data Transmission: The UE App/OS tags the packet with a 16-bit CAH.

RAN Handling: The gNB decrypts the PDCP layer and reads the CAH at a fixed offset to adjust scheduling (e.g., extending C-DRX for an AI text stream).

Core Handling: The UPF extracts the CAH, determines the routing path (e.g., selecting a low-latency Edge/MEC route), and strips the 2-byte header.

Dual Visibility Lifecycle: The header is read non-destructively by the RAN for scheduling but is explicitly stripped by the UPF before leaving the 6G Core, ensuring "Middlebox Safety" for the external Data Network.

Mobility Fallback: If the UE hands over to a legacy gNB, it immediately transitions to a "Legacy State," inhibiting CAH insertion to prevent packet corruption at non-supporting nodes.

#### 6.X.Y.1.1 Context-Aware Header Details

#### A. Protocol Stack with CAH

This figure illustrates the architectural placement of the Context-Aware Header (CAH) within the 6G user plane protocol stack.

Adaptation Layer Placement: The CAH is positioned as a "shim" layer strictly between the User IP layer and the 3GPP SDAP layer.

Decryption Boundary: It is located below the IP layer but above the SDAP/PDCP layers, ensuring that after PDCP decryption at the gNB, the metadata is immediately visible in cleartext to the scheduler.

Hardware Efficiency: By residing at the "head-of-line" of the SDAP Service Data Unit (SDU), it allows for O(1) deterministic parsing by gNB ASIC/FPGA hardware without requiring deep packet inspection.

**![](data:image/png;base64...)**

**Figure** A**: Protocol Stack with CAH**

#### B. The End-to-End Transport path:

This figure depicts the end-to-end (E2E) lifecycle of a context-aware packet, demonstrating the "Dual Visibility" principle where the RAN and Core take distinct actions based on the same metadata.

**UE Insertion**: The UE (via OS API or Modem heuristics) tags the uplink packet with a 16-bit CAH indicating the application intent (e.g., GenAI Prompt).

**RAN Awareness**: The gNB decrypts the PDCP layer and reads the **Workload Tag** to perform **Predictive QoS**. It may extend C-DRX timers or pre-allocate grants for the anticipated downlink response.

**Core Routing**: The UPF identifies the **Service Category** and selects the optimal routing path, such as an **Edge/MEC offload** for latency-sensitive XR or AI inference.

**Header Stripping**: The UPF removes the 2-byte CAH before forwarding the packet to the **N6 interface**, ensuring the external Data Network receives a standard, unmodified IP packet.

**![](data:image/png;base64...)**

**Figure B: The End-to-End Transport paths**

#### C. Anatomy of 16-bit header (CAH)

This figure provides the bit-level mapping of the CAH, which enables the system to categorize, predict, and quantify traffic needs.

Service Category (Bits 15-13): Identifies the high-level application domain, such as 001 for GenAI, 010 for XR/Gaming, or 011 for V2X.

Workload Tag / Downlink Predictor (Bits 12-8): Signals the *expected* traffic pattern of the response. Examples include 00001 (Text Stream), which predicts a "trickle" pattern, or 00010 (Image Gen), which predicts a massive single burst.

Context Quantifier (Bits 7-0): Provides a context-dependent scale. For gaming, this may represent a Latency Budget (e.g., 10ms); for image generation, it represents an Estimated Magnitude to help the gNB size the radio grant.

#### 6.X.Y.2 Procedures

Editor's note: This clause will describe the high-level procedures and information flows for the solution.

This figure illustrates the high-level signaling and data flow for a context-aware PDU session, enabling predictive QoS and intelligent routing.

![](data:image/x-emf;base64...)

**Figure 6.5.X.2.1: End-to-End Context Aware Traffic Flow and Procedures**

Step 1, UE generate data: The UE Application generates an encrypted data packet, such as a Generative AI prompt or a latency-critical Gaming input.

Step 2, UE insert Context-Aware Header (CAH): The UE (App, OS, or Modem) inserts the 16-bit CAH shim between the User IP layer and the 3GPP SDAP layer. The header indicates the Service Category (e.g., GenAI) and the Workload Tag (e.g., Image Generation) which acts as a predictive indicator for the expected downlink response.

Step 3, UE Uplink transmission: The UE transmits the encrypted Radio PDU, now containing the cleartext CAH shim, to the gNB over the air interface.

Step 4, RAN inspect CAH (RAN Awareness): After PDCP decryption, the gNB (RAN) performs a non-destructive read of the CAH at a fixed offset. Recognizing the Workload Tag (e.g., "Image Generation" predicts a massive burst), the gNB proactively adjusts scheduling, such as pre-allocating large downlink grants or extending C-DRX timers to maintain connectivity for a "Text Trickle".

Step 5, gNB/RAN forward via GTP-U to UPF: The gNB (RAN) forwards the packet to the UPF through the GTP-U tunnel, keeping the CAH intact within the payload to enable Core-side awareness.

Step 6, UPF read CAH and select path (Core Awareness): The UPF receives the packet and parses the payload start to read the CAH. Based on the Service Category and metadata, the UPF executes intelligent routing decisions, such as selecting a low-latency Edge/MEC route or a specific Network Slice.

Step 7, UPF strip CAH: To ensure complete compatibility with standard internet infrastructure, the UPF strips the 2-byte CAH from the payload, restoring the packet to its standard IP format.

Step 8, UPF forward Standard IP Packet: The UPF forwards the clean, standard IP packet to the Application Server via the N6 interface, ensuring the signaling remains strictly internal to the 6G System and is "Middlebox Safe".

#### 6.X.Y.3 Impacts on Services, Entities and Interfaces

Editor's note: This clause captures impacts on existing services, entities and interfaces.

**UE:** Requires the ability to insert a shim header between the IP and SDAP layers based on application hints or modem heuristics.

**gNB/RAN node:** Requires a hardware/firmware update to parse a fixed 2-byte offset in the SDAP SDU non-destructively.

**UPF:** Requires the ability to extract and strip the CAH from the user plane payload.

**SMF/PCF:** Enhanced to authorize CAH usage based on subscriber profile and application requirements.
