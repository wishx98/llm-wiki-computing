### 6.22.2 Solution #22.2: Coordination between Communication and Computing is controlled by 6G CN

#### 6.22.2.0 Topics addressed and High-level Solution Principles

**NOTE (from Pen holder):** This solution category is assumed to address bullet#1b, 3a, 3c, 3d, 3e, 4a, the missing aspect can be added later.

This solution addresses bullet #1b, 3a, 3c, 3d, 3e, 4a under KI#22 since the solution for these 6 aspects have dependency with each other, and they are normally addressed in a single solution.

This solution is extracted from Solutions #1, 3, 5, 6 (Discovery and Selection of Computing Sites), 7, 8, 10, 11, 13, 15 (Computing Request over CP), 16, 21, 25, 26 (DNS query procedure), 33(UE Requested part), 34, 36, 38, 39, 40 of Annex X.22.

There are following variants:

UE Requested Computing Support in 6G by using SM NAS (Solutions #1, 8, 13, 15 (Computing Request over CP), 21, 25, 26 (DNS query procedure), 33(UE Requested part), 34 of Annex X.22.)

UE Triggered Computing Service Support in 6G by using SM NAS and UP connectivity to a Computing Site in the SHE, an Operator controlled domain other than 6G CN (Solution #7 of Annex X.22)

UE Requested Computing Support in 6G by using Computing NAS for Computing Site connecting to 6G UPF (Solution #3, 5, 10, 11, 15, 26 (Iaas) 36/38/39/40 in Annex X.22.)

UE Requested Computing Support in 6G by using Computing NAS for Computing Site connecting to 6G Data Plane Function (Solution 11 (over DP) and 15 (Computing Request over DP) in Annex X.22)

UE Requested Computing Support with Computing messages over User Plane (Solution #14, 15, 23 in Annex X.22.)

UE Requested Computing Support in 6G over new Plane (e.g., service plane) (Solution # 16 and 26 (only the Operator Service with NF part) in Annex X.22)

UE Initiated Computing Service Support and SMF Controlled Coordinated selection of UPFs and compute sites (Solution #35 in Annex X.22)

AF Requested Computing Support via CCF (Solution #**5, 10,** 16, 17, 33 in Annex X.22)

The solution illustrates how the UE or AF requests the computing support from 6G CN and how the 6G CN coordinates the communication and computing requirements and select the proper Computing Site(s) for the UE’s PDU Session.

The solution for variants A/C/D/E/F/H is based on the following principles:

In control plane there is an NF (e.g., 6G PCF, 6G SMF, CMF, etc.) is used to discover the suitable Computing Node (which is an entity that can represent Computing Site) for the computing request from UE.

The CMF (Computing Management Function) is aware of the status and capabilities of the internal/external computing resources via the Computing Node/Site. The 6G SMF and CMF need to consider the coordination between the computing and communication to do the selection of Computing Node/Site.

The communication between the UE and Computing Node/Site for Computing Service is via user plane.

The coordination between the Computing metrics and Communication QoS parameters on computing session is determined by a CP NF. The CP NF determining Computing metrics and QoS related information (e.g., PDB, Computing latency) based on the subscription and Computing resources status.

NOTE: The coordination between the Computing metrics and Communication QoS parameters on computing session may need to coordinate with KI#5 (e.g., on adaptive QoS, QoS monitoring).

The solution for variant H is based on following principles:

**AF trigger computing session request**
The AF determines the need for computing services based on application logic and requests computing support using abstracted capability (e.g., CCC) and service constraints (e.g., latency).

**Abstraction of computing resources**
The network abstracts heterogeneous computing resources into pre-defined computing capability classes (CCC) and exposes them to the AF, avoiding direct exposure of infrastructure details.

**Network-controlled coordination**
The network instantiates appropriate computing resources based on the AF’s computing session requirements and operator policies, and updates the computing session context to the PCF.

**Policy-based binding**
The computing session is associated with the PDU session via policy control, enabling coordinated handling of communication and computing flows.

**Closed-loop adaptation**
The network monitors service conditions and may reselect computing resources or adjust configuration to maintain service performance.

**API-based exposure to third-party applications**
Computing capabilities are exposed via standardized APIs, allowing authorized third-party applications (AFs) to request and utilize computing services in an open and interoperable manner.

#### 6.22.2.1 Description

##### **6.22.2.1.1 Architecture and General Description**

6.22.2.1.1.1 Description for variants A/C/D/E/F/H

A Computing Management Function is introduced to monitor the computing resources in 6G CN, the Computing resources are assumed to be located in Computing Site and accessed via 6G UPF. The Computing Site can be located in DN or 6G CN. When the Computing Site is located in 6G CN, it can be separated from or collocated with 6G UPF. Figure 6.X.Y.1.2-1 shows the architecture.

The computing node is introduced to route the Computing data packet between 6G UPF and Computing Site, represent the computing site (in the 6G Core Network or in the Data network) to monitor the Computing resources in Computing Site(s), receive and execute the Computing resource Request from CMF.

The CS (Computing Site) is assumed to provide various services based on different types of computing resources. The computing site in CN is either collocated with or separate from computing node. The interface and protocol between the CMF and Computing Node will be standardized, while the interface between Computing Node and Computing Site will not be standardized (e.g., via k8s) in 3GPP SA2.

6.22.2.1.1.2 Description for variant H

In this solution, communication and computing are coordinated under network control while maintaining clear separation of responsibilities.

The UE establishes the PDU session for connectivity. The AF determines computing requirements. Based on the exposed CCC Profiles (CCCP), the AF selects a preferred CCCP and provides service-level constraints, such as maximum latency. The AF then sends a computing session request to the CCF, expressing the desired computing capability and performance requirements in an abstract manner.

The CCF determines the appropriate computing resources by considering the AF’s service requirements, user subscription information, and operator policies, and subsequently requests the instantiation of the selected computing resources from the SHE-C.

The CCF requests the PCF to register the computing session information. The PCF associates the computing session with the corresponding PDU session through policy-based binding and distributes the necessary information to the UE and the UPF, thereby enabling traffic steering toward the selected computing resources and ensuring QoS enforcement.

The network may adapt to changing conditions by updating the computing session and reselecting resources, ensuring service continuity.

The detailed workload partitioning remains at the application layer.

##### 6.22.2.1.2 Coordination between Communication and Computing

6.22.2.1.2.1 Description for variants A/C/D/E/F/H

The coordination solution for variants A/C/D/E/F/H comprises:

UE/AF requests computing support from 6G CN by using the Computing Request including the computing related information (e.g., Computing Request indication, Task type, overall Task latency, overall Task throughput, Computing resource type, RTT delay requirements, etc.) in order to satisfy the service experience.

Based on the UE/AF request, authorization information in the User Subscription data and Computing Policy (e.g., locally configured or from PCF), the CP Function (e.g., 6G SMF, CMF, CCF) derives the QoS profile for RAN and Computing Enforcement rule for Computing Node/Site for Computing Service by coordinating with CMF which selects proper Computing Site(s).

Computing Node/Site for Computing Service enforces the Computing Enforcement rule received from the CP Function by allocating corresponding computing resources of a Computing Site or routing the data packet(s) to the selected Computing Node/Site.

Editor’s note: The detailed computing related information from UE can be captured in the detailed procedures.

The coordination NF variants:

6G SMF derives the communication QoS parameters and Computing requirements (e.g., latency, throughput, resource type, etc.)

CMF derives the communication QoS parameters and Computing requirements (e.g., latency, throughput, resource type, etc.)

6G PCF derives the communication QoS parameters and Computing requirements (e.g., latency, throughput, resource type, etc.)

The coordination decision on how to derive the Computing related parameters in the Computing Enforcement rule in Computing Node/Site can be made based on UE requested information, authorization in User subscription and PCC rule/Computing Policy.

6.22.2.1.2.2 Description for variant H

The proposed solution introduces a network-controlled coordination framework where communication and computing are treated as distinct but tightly coupled domains. The UE is responsible only for establishing the PDU session, while the need for computing is determined by the AF based on service logic. The AF expresses its intent using an abstracted capability (CCC) and service constraints such as latency and requests a computing session via API. Based on this intent, the network selects appropriate computing resources by jointly considering communication and computing conditions, without exposing underlying infrastructure. Coordination is realized through policy-based binding between the PDU session and the computing session, enabling consistent handling of traffic through steering and QoS enforcement. The network can dynamically adapt to changing conditions, such as mobility or performance degradation, by updating the computing session and reselecting resources, ensuring service continuity. By leveraging CCC-based abstraction, AF-driven control, and policy-based coordination, the solution achieves efficient, scalable, and operator-controlled integration of computing while maintaining a clear separation between application logic and network functions.

6.22.2.1.5 Computing Node/Site Discovery and (Re-)Selection

For variants A/C/D/E/F/G/H, one of the following NFs is proposed to perform the Computing Node/Site discovery and (re-)selection:

6G SMF;

CMF;

NRF.

For variant H, the proposed solution performs computing node (or site) discovery and (re)selection under network control, with the CCF acting as the central decision entity. Instead of relying on the UE or AF to select specific resources, the CCF collects available computing resource information from SHE-C and abstracts it into Computing Capability Classes (CCC). Based on the CCC selected by the AF and service constraints such as latency, the network identifies suitable candidate computing sites by jointly considering both communication and computing conditions, including UE location, network state, and resource availability. This abstraction-based approach allows efficient and consistent resource selection without exposing underlying infrastructure details. For ongoing services, the same mechanism enables fast and dynamic re-selection: when conditions change due to mobility, resource degradation, or congestion, the AF may update the requested CCC, and the network can immediately re-evaluate and select an alternative computing site. As a result, the solution supports latency-aware, policy-driven, and secure resource selection while maintaining service continuity and minimizing system complexity.

1. UE Registration and PDU session establishment
The UE establishes connectivity with the network.

2. Application layer communication between UE and AF
(Out of scope) UE report device computing capability and current state of computing resource and power. AF and UE exchange data about computing workload.

3. Computing Offloading decision and Computing resource selection
The AF decide for computing offloading configuration. The AF retrieves computing class profiles and selects an appropriate computing resource with proper CCC and maximum latency requirement.

4. Computing Session request
The AF sends a computing service request to the CCF including the preferred CCC Profile, max latency value. If AF is 3rd party application in outside of operator network, request go to NEF and NEF forward to CCF.

5. Authentication and authorization
The CCF perform authentication and authorization for the requesting User and requested computing service.

6-8. Computing resource instantiation request
⑥ The CCF decide proper Computing site based on the AF preference, requirement and operator policy. CCF requests the SHE-C to instantiate computing resources on the selected computing site.
⑦ (Out of Scope) Computing environment preparation
The SHE-C allocates computing resources and prepares the execution environment on the selected SHE-CN.
⑧ The SHE-C notifies the CCF that the computing resources and execution environment are ready.

9. Computing Session Update
The CCF send Computing Session Update to PCF

10. Binding Computing Session context and PDU session context

11. Distribute the updated context to UE and UPF
PCF bind the Computing Session Contexts into PDU session context. The context may include computing node address, AI workload information, computing class context, latency requirements (Max Latency). PCF decide that it must be updated to UE and UPF and distributed it to UE and UPF. UE and UPF provision the context.

12. Application layer communication between UE, SHE-CN and AF
(Out of scope) UE, Computing node and AF share the detail workload sequence and data type that they exhange
