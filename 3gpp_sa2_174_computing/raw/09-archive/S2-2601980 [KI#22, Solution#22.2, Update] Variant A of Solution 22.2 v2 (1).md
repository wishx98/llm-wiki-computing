# 1 Introduction

This is a solution update based on S2-2600417, S2-2600410 and S2-2601619 at SA2#173 to accommodate the approved the structure and clause numbers for KI#22.

SA1 has defined several computing use cases with the requirements on computing resource selection, provision, and management. A typical scenario is computing task offloading (e.g., XR rendering, and LLM inference), in which the UE requests the computing site in 6G network to provide computing service support for data. As shown in the Figure-1 below, the overall task latency comprises communication delay and computing delay.

![](data:image/png;base64...)

**Figure 1: Communication delay and Computing delay**

The Computing Site can be either separate from the User Plane Function via 6G9 or 6G6 interface or collocated with User Plane Function:

**Option-1:** The Computing Site is separate from User Plane Function, then based on Computing Policy received from CP Function, the User Plane Function allocates computing resources in Computing Site(s) via API (e.g., K8s APIs), and monitors the usage of computing resources in the Computing Site(s). In this option, it assumes “UPF + Computing Site” as a virtual Computing Node. The Computing Site can be located in 6G CN or DN.

**Option-2:** The Computing Site is collocated with User Plane Function, then based on Computing Policy from CP Function, the UP Function allocates the local computing resource and monitor the usage of computing resources. In this option, the collocated UPF and Computing Site is a Computing Node.

**For the communication delay,** it corresponds to traditional QoS parameters (e.g., PDB, data rate, etc.). The RAN node allocates the radio resource based on the QoS profiles received from CN. For the computing task offloading scenario, the round-trip communication latency between UE and the anchor UP Function should be considered.

**For the computing delay,** it is not considered in the traditional QoS design, while the delay will happen in the computing site and affects the overall delay requirement for a service requiring massive computing support at network. In case the computing site is collocated with UP Function, the computing delay can be measured by UP Function based on the implementation; otherwise, the UP Function can measure the computing delay (e.g., throughput, latency, etc.) either by itself (refer to Figure 2) or via APIs (e.g., K8s APIs) between the UP Function and the computing site.

NOTE: The 6G network such as UP Function can allocate the computing resources and get usage report by invoking the K8s APIs. With configuring K8s APIs and on-demand plug-in, the 6G network can control the computing resource on computing site as well as collecting the computing usage status, which could be targeting to a certain task for one or a group of UEs.

## 6.22 Solutions to KI#22

### 6.22.2 Solution #22.2: Coordination between Communication and Computing is controlled by 6G CN

#### 6.22.2.0 Topics addressed and High-level Solution Principles

**NOTE (from Pen holder):** This solution category is assumed to address bullet#1b, 3a, 3c, 3d, 3e, 4a, the missing aspect can be added later.

This solution is extracted from Solutions #1, 3, 5, 6 (Discovery and Selection of Computing Sites), 7, 8, 10, 11, 13, 15 (Computing Request over CP), 16, 21, 25, 26 (DNS query procedure), 33(UE Requested part), 34, 36, 38, 39, 40 of Annex X.22.

There are following variants:

UE Requested Computing Support in 6G by using SM NAS (Solution #1, 6 (collocating CMF with NRF), 8, 13, 15 (Over CP), 20, 21, 25, 33 (UE Requested), 34 of Annex X.22.)

UE Triggered Computing Service Support in 6G by using SM NAS and UP connectivity to a Computing Site in the SHE, an Operator controlled domain other than 6G CN (Solution #7 of Annex X.22)

UE Requested Computing Support in 6G by using Computing NAS for Computing Site connecting to 6G UPF (Solution #3, 5, 10, 11, 15, 26 (Iaas) 36/38/39/40 in Annex X.22.)

UE Requested Computing Support in 6G by using Computing NAS for Computing Site connecting to 6G Data Plane Function (Solution 11 (over DP) and 15 (Computing Request over DP) in Annex X.22)

UE Requested Computing Support with Computing messages over User Plane (Solution #14, 15, 23 in Annex X.22.)

UE Requested Computing Support in 6G over new Plane (e.g., service plane) (Solution # 16 and 26 (only the Operator Service with NF part) in Annex X.22)

UE Initiated Computing Service Support and SMF Controlled Coordinated selection of UPFs and compute sites (Solution #35 in Annex X.22)

AF Requested Computing Support via CCF (Solution #5, 10, 16, 17, 33 in Annex X.22)

The solution illustrates how the UE or AF requests the computing support from 6G CN and how the 6G CN coordinates the communication and computing requirements and select the proper Computing Site(s) for the UE’s PDU Session.

The solution for variants A/C/D/E/F/H is based on the following principles:

In control plane, an NF (e.g., 6G PCF, 6G SMF, CMF, etc.) is used to discover the suitable Computing Node (which is an entity that can represent Computing Site) for the computing request from UE.

The CMF (Computing Management Function) is aware of the status and capabilities of the internal/external computing resources via the Computing Node/Site. The 6G SMF and CMF need to consider the coordination between the computing and communication to do the selection of Computing Node/Site.

The communication between the UE and Computing Node/Site for Computing Service is via user plane.

The coordination between the Computing metrics and Communication QoS parameters on computing session is determined by a CP NF (i.e., SMF, CMF). The CP NF determines Computing metrics and QoS related information (e.g., PDB, Computing latency) based on the UE requested overall latency/throughput, subscription and Computing resources status.

NOTE: The enhancement of Communication QoS parameters on computing session may need to coordinate with KI#5 (e.g., on adaptive QoS, QoS monitoring).

The solution for variant B is based on the following principles:

At PDU session establishment, the UE indicates the PDU session is for a Computing Service.

The SMF selects a CCCE that performs CC policy association with the PCF. The CC profile can be pre-configured or optionally provisioned by AF request.

The selected UPF provides a CS ID (Computing Site ID) associated to the UPF to the SMF, which forwards it to the CCCE. The CS ID is used by the CCCE to identify the SHE controller to be contacted.

The CCCE queries the SHE controller (which is out of 6G CN) to retrieve available resources and determine the CC policy applicable to the Computing Service session.

The CCCE enforces the SM policy toward the SMF and the CM policy toward the SHE controller by requesting reservation of the required SHE resources for the computing Site.

The solution for variant G is based on following principles:

Avoid any additional requirements on HLOS in the UE and UE applications and ensure that any 3GPP enhancements for compute can work with different computing/virtualization frameworks.

The UE application interacts with a computing framework or a network-side application component that uses a computing framework layer in the user-plane.

An AF (representing a computing framework or a network-side application component that uses a computing framework) that is deployed in the service hosting environment may provide the maximum acceptable aggregated latency (the sum of the communication latency (N3 latency and N6 latency) and the computing latency) and information about the computing latency of available computing sites for a particular UE (i.e., UE-related compute task) to the network.

Based on AF influence, the 6G SMF:

selects a DNAI and computing site for which the communication latency (N3 latency and N6 latency) and the computing latency for the computing site as provided by the AF is lower than the maximum acceptable aggregated latency.

adds a ULCL / BP and L-PSA for the selected DNAI;

adjusts the QoS parameters of the flows between the UE and the computing site;

informs the AF about the selected DNAI and computing site.

#### 6.22.2.1 Description

##### **6.22.2.1.1 Architecture and General Description**

###### 6.22.2.1.1.1 Description for variants A/C/D/E/F/H

A Computing Management Function is introduced to monitor the computing resources in 6G CN, the Computing resources are assumed to be located in Computing Site and accessed via 6G UPF. The Computing Site can be located in DN or 6G CN. Figure 6.X.Y.1.2-1 shows the architecture.

The computing node is introduced to route the Computing data packet between 6G UPF and Computing Site, represent the computing site (in the 6G Core Network or in the Data network) to monitor the Computing resources in Computing Site(s), receive and execute the Computing resource Request from CMF. The computing node can be collocated with UPF.

The CS (Computing Site) is assumed to provide various type of services or service components (such as XR rendering, LLM inference) based on computing resources. The interface and protocol between the CMF and Computing Node will be standardized in 3GPP SA2, while the interface between Computing Node and Computing Site will not be standardized (e.g., via k8s) in 3GPP SA2.

**Figure 6.22.2.1.2-1(Option #1): 6G Computing Architecture with Computing Site in 6G CN**

NOTE 1: In Option #1, the Computing Site in CN can be collocated with Computing Node.

![](data:image/x-emf;base64...)

**Figure 6.22.2.1.2-2(Option #2): 6G Computing Architecture with Computing Site in DN**

NOTE 2: In both Option #1 and Option #2, the Computing Node can be collocated with 6G UPF.

The Computing Site can be located in DN or 6G CN. When the Computing Site is located in CN, it can be either separated from the 6G UPF or collocated with 6G UPF.

NOTE 1: There two alternative options to place CMF, one is collocating CMF with 6G SMF, the other is to collocate CMF with NRF.

NOTE 2: The CMF may collect the Computing resource related information (e.g., resource type, resource status, etc.) from Computing Site directly or via 6G Anchor UPF.

NOTE 3: An Intermediate UPF may be added between 6G RAN and 6G Anchor UPF.

NOTE 4: Whether the UE’s Computing Request will go through an AI Agent Function will be coordinated with KI#1.1 (NAS) and KI#18 (AI for 6G architecture).

NOTE 5: Whether the interfaces (e.g., between the 6G-SMF and 6G UPF, between CMF and 6G UPF, between CMF and Computing Site, etc.) are service based will depend on the conclusion on KI#2 (SBA).

NOTE 6: The 6G UPF and Computing Site can be regarded as a virtual Computing Node which can be controlled by both CMF and 6G SMF.

NOTE 7: The UPF and Computing Node can be collocated and become a Data Plane Function as proposed in variant D, this aspect will be coordinated with KI#21.

NOTE 8: Whether a new Plane (e.g., Service Plane) is used between the UE and the CMF will be coordinated with KI#1.2.

NOTE 9: Whether NAS is terminated at 6G AMF and whether there is interface between 6G RAN and 6G AMF depend on the conclusion for KI#1.1.

NOTE 10: In some options, the Application Server in DN is not required.

NOTE 11: Whether the interface between CMF and Computing Node/Computing Site is service based or not should be explicitly described in specific solution variants.

NOTE 12: In some options, the Computing Node is not introduced, the packet routing is between 6G UPF and Computing Site.

##### **6.22.2.1.2 Coordination for Communication and Computing**

###### 6.22.2.1.2.1 Description for variants A/C/D/E/F/H

The coordination solution for variants A/C/D/E/F/H comprises:

UE/AF requests computing support from 6G CN by using the Computing Request including the computing related information (e.g., Computing Request indication, Task type, overall Task latency, overall Task throughput, Computing resource type, RTT delay requirements, etc.) in order to satisfy the service experience.

Based on the UE/AF request, authorization information in the User Subscription data and Computing Policy (e.g., locally configured or from PCF), the CP Function (e.g., 6G SMF, CMF, CCF) derives the QoS profile for RAN and Computing Enforcement rule for Computing Node/Site for Computing Service by coordinating with CMF which selects proper Computing Site(s). The derived QoS profile and Computing Enforcement rule are used to fulfil the overall task latency and overall task throughput for a task as requested by UE/AF.

Computing Node/Site for Computing Service enforces the Computing Enforcement rule received from the CP Function by allocating corresponding computing resources of a Computing Site or routing the data packet(s) to the selected Computing Node/Site.

Editor’s note: The detailed computing related information from UE can be captured in the detailed procedures.

The coordination NF variants:

6G SMF derives the communication QoS parameters and Computing related parameters (e.g., computing latency, throughput, resource type, etc.)

CMF derives the communication QoS parameters and Computing related parameters (e.g., computing latency, throughput, resource type, etc.)

6G PCF derives the communication QoS parameters and Computing related parameters (e.g., computing latency, throughput, resource type, etc.)

The coordination decision on how to derive the Computing related parameters in the Computing Enforcement rule in Computing Node/Site can be made based on UE requested information, authorization in User subscription and PCC rule/Computing Policy.

##### 6.22.2.1.3 Computing Node/Site Discovery and (Re-)Selection

For variants A/C/D/E/F/G/H, one of the following NFs is proposed to perform the Computing Node/Site discovery and (re-)selection:

6G SMF;

CMF;

NRF.

For variant B, there is no selection of the Computing Site, which is determined by the CS ID associated to the selected UPF. Selection of the computing resources performed by the SHE Controller is out of scope of SA2.

###### 6.22.2.1.3.X Variant A

**Initial Selection:**

When receiving the Computing support request for a UE’s application for a PDU Session from a 6G SMF, the CMF selects a suitable Computing Node based on the availability status of the requested Computing resources (e.g., Processor Unit type, the according available Computing resources in a Computing Node) which can serve the UE’s location and the requested PDU Session. The CMF can respond to the 6G SMF with the IP address and tunnel information of the selected Computing Node, then 6G SMF configures the RAN/6G UPF and Computing Node with each other’s tunnel information.

In case the Computing Site is separated from the UP Function, the UP Function can select a proper Computing Site and respond to 6G SMF with the IP address of the Computing Site.

**Re-selection:**

In case there is status change (e.g., computing burden overloading) of the Computing Node supporting a UE’s application based on the report from the Computing Node, the CMF may decide to select another Computing Node which can serve the UE’s location. The CMF can interact with 6G SMF to update the change of the Computing Node (e.g., the new IP address), then 6G SMF can re-configure the RAN/6G UPF and the Computing Node for the IP address and tunnel information for each other.

In case the Computing Site is separated from the UP Function, the UP Function can select a proper Computing Site and respond to 6G SMF with the IP address of the Computing Site.

In case the UE needs to be aware of the IP address of Computing Site serving a PDU Session after the (re-)selection, and the UE is informed the (re-)selected Computing Site by the 6G SMF.

In case the 6G-SMF and CMF are collocated, the interaction between 6G-SMF and CMF is not needed.

#### **6.22.2.2 Procedure**

**6.22.2.2.X Procedures for Variant A**

**6.22.2.2.X.1 UE Triggered Computing Request Procedure**

![](data:image/x-emf;base64...)

**Figure 6.22.2.2.X.1-1: UE Triggered Computing Request procedure**

1. UE sends Computing Request (e.g., PDU Session Establishment/Modification Request) with required Computing information (e.g., Task Type, Overall Task throughput, Overall Task latency, etc.) to the CP Function.

2. Computing authorization check between 6G-SMF and UDM.

3. Computing Policy retrieval from 6G-PCF.

4. Based on the required Computing information and Computing Policy, the CP Function determines the address of selected Computing Site, QoS parameters and Computing Enforcement rule. The The PDR defined in TS 23.501 can be used to detect a data packet for computing.

5. The CP Function provides Computing Enforcement rule (e.g., Resource type, Computing throughput, Computing latency, Priority, etc.) to the User Plane Function.

6. The User Plane Function allocates the computing resource (e.g., by invoking K8s API) in the Computing Site serving it.

7. The User Plane Function responds to the CP Function including the IP address of Computing Site.

8. The CP Function sends a 6G2 Request to RAN including the 6G2 information (e.g., QoS profile, etc.) and the NAS Message (e.g., Computing Response including the IP address of Computing Site, etc.).

9. The AN node performs radio resource configuration based on 6G2 information and QoS profile. AN node forwards the NAS Message received in step 8 to UE.

NOTE 1: This procedure can be used for initial Computing request and later Computing update.

Editor’s note: other procedures (e.g., Computing Release, AF triggered procedures, etc.) are FFS.

**6.22.2.2.X.2 Computing Node (re-)selection procedure**

![](data:image/x-emf;base64...)

**Figure 6.22.2.2.X.2-1: Computing Node (re-)selection procedure**

1. 6G UE may send the Computing Site (re-)selection Request to 6G SMF for a specific Computing task. The detailed parameters included in this request will be defined by solution for KI#22.1b.

2. The 6G SMF sends the request to CMF to (re-)select a suitable Computing Node. How the 6G SMF determines the detailed parameter included in this request will be defined by solution for KI#22.3a.

3. CMF responds to the 6G SMF about the (re-)selected Computing Node information (e.g., IP address).

4. The 6G SMF configures the RAN tunnel information towards the (re-)selected Computing Node (i.e., 6G UPF) for data routing. The UP Function in the Computing Node selects the proper Computing Site and includes the IP address of the Computing Site in the Response to 6G SMF.

5. The 6G SMF configures the RAN node with the (re-)selected Computing Node.

6. If the (re-)selection procedure is triggered by 6G UE, the 6G SMF will respond to UE about the IP address of the selected Computing Site.

#### **6.22.2.3 Services, Entities and Interfaces**

**6.22.2.3.X Variant A**

**UE:**

Provide required Computing information (e.g., Task Type, Overall Task throughput, Overall Task latency, etc.) to CP Function.

Receive Computing Response from CP Function.

Send the Computing Site (re-)selection Request to 6G SMF;

Receive and store the (re-)selected Computing Site.

**6G-SMF:**

Retrieve subscription information from UDM.

Retrieve Computing Policy from 6G-PCF.

Receive required Computing information from UE.

Coordinate between PDB and Computing latency.

Coordinate between the data rate and Computing throughput.

Derive the QoS profile and send to RAN.

Derive the Computing Enforcement rule and send to UP Function, configure the UP Function for computing resource support.

Send Computing Response to UE.

Retrieve the IP address of a selected Computing Node from CMF;

Configure the RAN node and Computing Node with each other’s tunnel information.

Receive the IP address of a selected Computing Site and forward to UE.

**CMF:**

Manage Computing Nodes.

Select the Computing Node and send Computing Node information (e.g. IP address) to 6G-SMF.

Re-select a suitable Computing Node based on the monitoring of the status of Computing resources of the Computing Nodes under its control.

**Computing Node:**

Receive and enforce the Computing Enforcement rule from 6G-SMF.

Allocate computing resource of a Computing Site locally or by invoking APIs.
