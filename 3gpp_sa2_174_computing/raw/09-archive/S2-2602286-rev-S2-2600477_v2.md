# 1. Introduction

This solution proposal is contributing to the solution variant 22.3 as listed in Table 6.22.0-1 of TR 23.801-01 v0.4.0. The updates in this version (with tracked changes) are proposed enhancements to 22.3 and 22.10 as captured in the endorsed pen-holder document S2-2601619 from SA2#173.

This solution addresses the Key Issue #22 and describes scenarios and architectural requirement for 6G Compute. The solution also proposes definition for compute resource in the context of 6G Compute.

During the discussions about the scope of 6G computing, a pertinent question was whether application clients /operating systems/devices can autonomously decide and offload compute tasks.

S2-2508649 put forward the following view of computing in 6G:

a) The UE may decide to move an application module into MNO’s cloud due to battery limit, processing power, user experience degradation, etc. and request for it.

b) Application logic allows for work split across the UE, MNO’s cloud and the AS (that may reside in service provider’s edge/cloud).

c) Work split between the UE and the network operators’ cloud can be controlled by the UE

However in the scenario above it is not accurate to say that the UE may decide to move an application module to cloud. This depends on the application design. The UE (especially, the MT part) may not be aware of the architecture used by an application installed on the device. Also, the work split between the client device and the cloud compute site may not be under the control of the UE. This also depends on the application framework in use.

A few contributions referred to various distributed computing architectures available today in the ecosystem. S2-2510052 referred to the IaaS (e.g., Juice, GPU-over-IP framework catering to CUDA APIs) and PaaS (e.g., Nvidia AI enterprise) architectures. S2-2508574 refers to various scenarios for IaaS and PaaS frameworks.

This contribution analyses the compute offload frameworks and puts forward architectural requirements for 6G system, definition of compute resource and presents the overall system architecture view.

It can be seen in Figure 1-1 that the compute offload architecture for GPU-over-IP rely on a framework that has a client side component (Juice Client) and server side component (Juice Agent) <https://docs.juicelabs.co/docs/juice/intro> . The client side component enables application clients to request for offloading tasks.

Though some of these frameworks claim to work agnostic to the application design, it comes with a performance penalty (https://docs.juicelabs.co/docs/juice/user-guide/performance-latency/example). This shows that if UE autonomously decides to offload an application components that is not designed for execution in a distributed manner, it can have a performance penalty. In a wireless network, the framework for offloading computing is a balancing act between device’s computing resources and network’s communication resources. Compute offload frameworks require applications to be specifically designed for distributed execution to be effective.

An offload of computing task works within this framework residing on both client and server.

Compute Architecture for Neural Networks (CANN) is a heterogeneous computing architecture, designed for AI with device-cloud consistency. CANN automatically identifies the computing capability of the running environment, and performs adaptive graph partition and collaborative device scheduling for the neural network.

Another paradigm in compute offload is the Server-less computing. Examples of this concept are Cloudflare’s Worker and Amazon’s Lambda. They are light-weight cloud services offering Function-as-a-service aimed to reduce the burden of managing server infrastructure in the backend. But this also reqiures that an application client is designed to use the service offered by these frameworks.

Another key aspect of the 6G system architecture for compute is concerning user privacy. Even though the compute tasks to offload and result of execution can be assumed to be protected end-to-end, the metadata from this procedure can lead to privacy concerns. Combining this metadata information with other UE identifiers enable profiling. It is desirable to define 6G system architecture such that a strict separation of information across the computing and communication domains can be achieved. An architectural requirement for user privacy in this context is proposed.

### 6.22.Y Solution #22.Y: Scenarios and architectural requirements for 6G compute

#### 6.22.Y.0 Topics addressed and High-level Solution Principles

This solution

- addresses Key Issue #22 bullet1, 2b, 3a

- proposes architectural requirement that 6GS shall enable compute task offloading agnostic to the specific application-layer protocol, allowing diverse computing frameworks to be deployed.

- proposes architectural requirement that 6GS shall support applying per-subscriber policies related to compute offload, while ensuring that subscriber information from network domain is not exposed to compute resources and computing task specific information from compute domain is not exposed to the network.

- defines compute resource as the application layer framework consisting of hardware and software components, that receives and processes a client’s request for offload of a computing task.

- describes the overall system architecture for compute in 6G.

- describes the concept of Correlation ID, to enable the 6G Core network functions to correctly associate different AF requests and take actions in a coordinated manner,

- describes exposure of network metrics consisting of predictive metrics and network routing context.

#### 6.22.Y.1 Description

This solution addresses the Key Issue #22 , bullet #1, 2b and 3a

##### 6.22.Y.1.1 Common scenarios for compute offload

Applications that rely on offloading tasks from a client device to a cloud based compute resource rely on a framework that has both client side component and server side component. The client side component enables application clients to request for offloading tasks. Some examples of these are Private Cloud Compute, Juice GPU-over-IP frameworks.

The application clients deployed on the UE may be following any of these architectures. The type of offload server suitable to that application vary depending on the application design. This information may not be visible to the UE (especially MT part of the UE).

It is not possible for the UE to autonomously decide about offloading compute tasks. The applications have to be specially designed to take benefit from the distributed computing architecture. If these frameworks are dealing with applications not designed for parallelism or distributed computing, they cannot function optimally.

The applications execute task offload to a site that is hosting a complementary server side software. The server side component helps to discover and/or to map the offloaded tasks to specific hardware (CPU, GPU) resources. The server side software is designed exclusively for the architecture of underlying harware.

Compute offload is a specialised service that needs to be supported by an application layer framework. In summary, it is not expected that 6G compute offload provides a generic compute offload capability for any type of application running on a UE.

One of the key features of the commonly used application layer frameworks is their accessibility from any type of network (cellular, non-cellular). These frameworks are closely related to the nature of applications (e.g., AI) and the hardware resources they require rather than the networks on which they are deployed. The interactions that enables application clients to execute a task offload is not foreseen to be in the scope of SA2 activities.

It is recommended that SA2’s aim should be to ensure an architecture that appeals to a broader set of applications/application developers. SA2 6G Computing architecutre should enable diverse types of application layer frameworks for compute offload to be deployed.

**Potential Architectural requirement 1:**

6G Compute: 6GS shall enable compute task offloading agnostic to the specific application-layer protocol, allowing diverse computing frameworks to be deployed.

##### 6.X.Y.1.2 Ensuring user privacy for compute offload

One of the key anticipated enhancements for compute architecture in 6G is to allow charging subscribers in both compute and communication domains. Acknowledging the importance to allow diverse charging models for MNO, it should also be noted that compute offload for processing workloads (e.g. AI) presents another unique challenge. A typical generative AI prompt may carry sensitive user information and it is essential to minimize exposure of such information to the 6G system. Even though the AI inference related communication can be assumed to be end-to-end encrypted, there is a potential for gathering information from the metadata shared at compute offload level (for example, pattern observed from the nature of tasks that are offloaded, location of the client).

The need to protect user privacy is not only relevant for AI workload offloading. This is applicable for any computing offload activity such that capability of combining metadata on the offloading task with other information about the UE (e.g., GUTI, Registration, Session Management) can be detrimental to subscriber privacy.

It is important to minimize the data collected about subscriber and also to avoid the possibility of correlating information from different domains (i.e., computing and communication).

**Compute characteristics profile**

6G CN and Compute Resource (either in SHE or external DN) may need to exchange information about the requirements of the task or capabilities of compute site to enable coordination of compute and communication. Example of such information include task characteristics (size, latency), compute resource availability, privacy (task or core isolation), security, priority. The nature of information exchanged need to ensure consistency across diverse type of applications. It is also important to ensure privacy in the context of this information exchange.

**Potential Architectural requirement 2:**

6GS shall support applying per-subscriber policies related to compute offload, while ensuring that subscriber information from network domain is not exposed to compute resources and computing task specific information from compute domain is not exposed to the network.

##### 6.22.Y.1.3 Definition of compute resource

The client side component of a compute offload framework (software that decides whether an offload has to be performed, discovers suitable offload resources and executes the offload) requires a compatible server software component to negotiate compute offload. It is not foreseen that any compute offload framework offers a capability for client side component to discover and prepare a server bare-metal configuration for offload.

Compute resource refers to application layer framework consisting of hardware and software components, that receives and processes a client’s request for offload of a computing task. It is not expected that 6GS is aware of the specifics of the computing framework and their interaction. The interaction between the client-side software and server side software is seen as application layer interaction for SA2.

From 3GPP perspective, the compute resource in 6G refers to an application layer framework that is ready for use by an application client on the UE which has the complementary framework.

##### 6.22.Y.2.2 Coordination between compute and communication by AF

The Application Function responsible for the compute and communication coordination invokes 6G CN services through the 6G Network Exposure Function APIs. The various services used by the AF may include requests for traffic routing, requests for QoS, requests for QoS monitoring and end-to-end delay measurements and network event subscriptions. To enable the 6G Core network to correctly associate different AF requests and perform actions (e.g., UPF (re-)selection considering N6 delay measurement and requested QoS) in a coordinated manner, the 6G Core network makes use of a Correlation ID. The Correlation ID is an operator-internal, session-scoped, non-persistent identifier that binds the network resources associated with the session (PDU session, QoS flows, mapping of externally exposed DNAI to UPF assignment) as known to the 6G Core network. The Correlation ID is unique per AF.

##### 6.22.Y.2.3 Network Metrics exposure to AF

The proposed interaction is based on the principle that the AF is the appropriate entity to coordinate compute and communication resource allocation, since the AF has direct knowledge of application-layer requirements and, through means outside the 3GPP system, maintains awareness of both the static capabilities and dynamic state of candidate compute resources. The role of the 6G Core network in this solution is to expose network-layer information that the AF cannot independently observe and that is necessary to translate its compute placement decisions into actionable 3GPP traffic influence requests.

In addition to the capability to expose current delay measured on N3, N9. N6 interfaces, the following information can be exposed by 6G CN.

Predictive metrics: NWDAF derived analytics output including predicted path latency, QoS sustainability estimates, congestion predictions with associated time horizons, and UE mobility trajectory forecasts where these affect the expected network path quality to a candidate CS. Predictive metrics shall be accompanied by a confidence indication and a validity period. The analytics may be enhanced with additional inputs e.g., N3, N9, N6 delay measurements, that are available to the 6G CN.

This network routing context constitutes information uniquely held by the 6G Core network and not independently observable by the AF. The exposure of network routing context is the primary enabler for the AF to translate its compute placement decisions into 3GPP traffic influence requests.

NOTE: The term "network routing context" as used here refers to a subset of CN-internal information that is topologically abstracted. Exposure is limited only to this subset that is necessary for the AF to formulate a traffic influence request.
